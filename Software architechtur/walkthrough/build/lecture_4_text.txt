Today’s Lecture
▶Today we continue the previous lecture’s theme by focusing
on testability and deployability as distinct quality attributes
• Testability is the “susceptibility to testing”2, and the
adjective “susceptible” means that something is “capable of
submitting to an action, process, or operation”3
• Deployability is not a well-recognized word according to
online dictionaries, but we can understand it to mean that a
developed system can be actually deployed for production with
a predictable and acceptable amount of time and effort (Bass
et al. 2021)

===== PAGE 6 =====
Testability (1/2)
▶Bass et al. (2021) characterize testability as “the ease with
which software can be made to demonstrate its faults”
• Note that their characterization frames testability to
software verification and not validation of requirements
• Know your history; it was Boehm (1984) who originally
made the distinction between software validation (“building
the right product”) and the verification (“building the
product right”) of the software developed
• Furthermore, their characterization assumes that a software
has at least one fault, which is theoretically a problematic
standpoint but still acceptable to convey the message

===== PAGE 7 =====
Testability (2/2)
▶Ammann & Offutt (2008, p. 284) motivate the importance of
testability with the following four points:
1. If testability is low, developers know that testing alone will not
do a good job at verifying; something else is needed
• But what that something else would even be?
2. Therefore, “any approach that improves testability will almost
certainly be worth the investment”
3. If testability is high, test results increase confidence about
software quality
4. “In cases where testability is low and resistant to improvement,
and where alternative means of verification are impractical, the
test engineer has a solid argument to take to management
that the artifact poses an undue amount of risk”

On the Stop Criterion
▶Even with a system with almost perfect testability, the
fundamental and unresolved issue concerning coverage and
stop criterion is present; when to stop testing?
▶Some heuristics (e.g., via coverage.py) could be used for
deducing an answer in practice
▶The question is classical in academic research; one way to
approach it is with statistical reliability engineering models,
such as those based on the non-homogenous Poisson process
▶Further note that tests cause maintenance overhead too

===== PAGE 13 =====
Testing (2/6)
▶There are many oracle variants (Barr et al. 2015):
1. Specified oracles are what we often thinking testing to be
about; 1 + 1 = 2, f (x) > 0 ∀x, and so forth and so on
2. Pseudo-oracles are often understood as alternative versions
developed independently, but, in theory, reference architectures
could also be seen as pseudo-oracles
3. Implicit oracles are something we know based on our general
software engineering knowledge; a crash of a system is clearly a
bug, a buffer overflow is clearly a bug, and so forth and so on
4. Derived oracles are something about right and wrong
behavior based on external artifacts (such as requirements,
protocols, standards, user feedback, runtime behavior, etc.)

Testability Tactics: Input and State (1/8)
▶The input generation tactic involves many automated
testing techniques beyond fuzzing, such as mutation testing
• When doing conventional unit testing and alike, one should
seek to design inputs to be sensible for verification, avoiding
trivial cases but covering boundary cases
• When operating with standards or protocols, seeking
100% coverage can be recommend with respect to what is
explicitly said in a given standard or a protocol

Testability Tactics: Input and State (2/8)
▶The record and play tactic conveys the important message
that test results should be reproducible
• This requirement may seem deceivingly simple when
thinking about unit tests and their oracles, but reproducibility
is one of the grand challenges, including with fuzzing and
nowadays also emerging LLM-based testing
▶When designing a testing framework, one should try to make
it record all states for a given input
• With distributed systems, one should also try to localize
storage for keeping a state of a whole system

Testability Tactics: Input and State (4/8)
▶Specialized interfaces are often needed for testing internals
• get() and set() for important variables, attributes, etc.
• report() for states, print debug() for verbose output,
and reset() for restoring a pristine state
• However, these also enlarge attack surfaces and may
confuse others, and thus conditional compilation (or
outsourcing to a specific opt-in component) is a good idea
#ifdef TESTING
assert(system.report() == STATE_SHUTDOWN)
if system.get_init() != 1:
system.exit(INIT_FAILED)
#endif

Testability Tactics: Input and State (5/8)
▶Assertions are a good way to verify runtime behavior
• They also improve testability; they usually output debug
information, fuzzers are very good at triggering them, etc.
▶Typically, one uses them to verify (a) inputs (preconditions)
and (b) outputs (postconditions), but sometimes it is useful
to also assert (c) intermediate states
▶They can be also extended for instance into:
ASSERT_FUTURE_STATE(10 seconds) == SYSTEM_SHUTDOWN
ASSERT_PROBABILITY(RANDOM_FOREST, test_set) > 0.7

Testability Tactics: Limit Complexity (1/3)
▶In general, limiting complexity contains many of the same
techniques that were discussed in the previous lectures; from
encapsulation to reducing dependencies
▶Therefore, what is more important to emphasize that both
systems and tests should ideally be deterministic
• However, they often are not for various reasons; there is a
certain stochastic element present in large systems
• For this reason, the NetBSD’s test suite uses an oracle that
has also flags for individual tests resembling cases such as
“expected failure due to a known extremely difficult bug” and
“sometimes fails in ARM64 and MIPS for unknown reason”

===== PAGE 29 =====
Testability Tactics: Limit Complexity (2/3)
▶To control the stochastic element, including with respect to
development processes, regression testing is commonly used
• In the context of software testing, a regression is something
that used to work but no longer works; hence, when fixing
such a bug, it is a good idea to add a regression test to
ensure the bug is noticed if it occurs again
▶Regression tests are also about documentation about a
system, which is also one way to limit complexity
• Without regression tests, do you think the NetBSD
developers would remember all bugs from the early 2010s?

Testability Patterns: Dependency Injection
▶Dependency injection means that a function or object
receives other functions or objects it requires (instead of
creating them itself)
▶It helps (unit) testing because also test instances can be
injected, and these can be further control and monitor a state
▶Overhead and small complexity increases are the trade-offs
def __init__(self, x) -> None:
self.__x = x
# <-- dependency injection

▶When using the strategy pattern, an abstract method is

===== PAGE 33 =====
Testability Patterns: Intercepting Filter Pattern
▶The intercepting filter pattern injects pre- and
post-processing filters to a function; these are executed before
and after the function’s own primary method is called
def __init__(self, pre, post):
self.__pre = pre
self.__post = post
def __f(self, x):
if self.__pre != None:
self.__pre(x)
z = self.__g(x)
if self.__post != None:
self.__post(z)

Deployability
▶Traditionally — and, depending on a domain, sometimes still
today, release engineering was a cumbersome process; a
release was planned and tested for a long time
▶From the late 2000s onward, so-called continuous software
engineering emerged, among them continuous integration
and continuous delivery
• A concept of rapid releases is also sometimes used
▶Recall the definition; it should be possible to deploy a system
with predictable and acceptable amount of time and effort
• This point underlines what was said earlier about QAs as
measurable or even falsifiable attributes
• Besides reduced time and effort, the continuous software
engineering practices have improved also predictability

===== PAGE 47 =====
Continuous Integration and Continuous Deployment
▶Continuous integration means that a system being
developed is continuously (e.g., daily) built and tested, such
as that a working version is always available
▶If a system is further deployed into production on a regular
basis (e.g., monthly), a further concept of continuous
deployment is often used
▶The notion of DevOps covers both (and possibly more)
▶These continuous practices require separating different
environments each other; e.g., “development environment
→integration environment →staging environment
→production environment”

===== PAGE 49 =====
Deployment Pipelines (2/4)
▶As a QA, deployability with a pipeline should be measurable
▶Bass et al. (2021) suggest to consider the following:
1. Cycle time: how long does it take to progress through a given
deployment pipeline (i.e., from the development to the
deployment environment)?
2. Traceability: is it possible to trace forward (from the
development to the deployment environment) and backwards
(from the deployment to the development environment)?
3. Repeatability: is the whole process reproducible; does it
always give the same outcome with the same artifacts?
• Easier said than done even with respect to builds alone!

Deployability Tactics: Manage Pipeline (2/4)
▶When operating with large systems, it is a good idea to scale
deploying incrementally to subsets of end-user population
• This practice is also known as canary testing
▶Sheehy & Reed (2025) recommend having a solid plan for
verifying deployment failures as well as for measuring and
documenting these with predefined health metrics
• Only when these indicate no problems for a subset of
end-users (or endpoints), a full roll-out should be done
• With particularly large systems, it may be reasonable to
define multiple subsets according to some criteria (e.g.,
different geographic locations, different configurations, etc.)

===== PAGE 56 =====
Deployability Tactics: Manage Pipeline (3/4)
▶For the scaling to be possible, deploying should be scripted
• Although some manual configuration is still typically
required, automate as much as possible
▶When health metrics indicate problems, it is important to
have a rollback functionality; it should be possible to revert
back to the previously deployed version of a system
• This functionality correlates with the traceability and
repeatability requirements; for a rollback to be possible, all
configurations and variability characteristics should be tracked
