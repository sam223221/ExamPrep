Performance and Scalability
▶Performance is about a system’s ability to meet resource
requirements; by following algorithmic complexity, we can
consider “time performance” (timing requirements) and
“space performance” (memory and disk use)
▶Performance is related to scalability, which is about
proportionally increasing performance by adding resources
▶When considering performance as a standalone QA, we are
interested in doing more with the resources we have, that is,
without adding new resources

Latency and Throughput
▶Particularly when considering networking, including with
respect to distributed systems, the two basic measures are:
1. Latency: the time to perform an action or to produce a result
2. Throughput: number of actions or results per unit of time
▶The optimum is maximal throughout with minimal latency

Performance Tactics: Resources 7→Concurrency (1/7)
▶Although dictionary definitions for the adjective concurrent
emphasize both things occurring at the same time
(simultaneous occurrence) and in parallel, we typically
separate concurrency from parallelism1
▶The reason is that parallelism can be seen to require multiple
CPUs, whereas concurrency can be done with a single CPU
▶In conventional programming with a single machine, it is
sensible to further simplify things by associating concurrency
with threads and parallelism with processes
• But note that the distinction is not accurate at the
operating system level due the emergence of multi-core CPUs

Performance Tactics: Resources 7→Copies of data (1/8)
▶For maintaining data redundancy, one must start from the
fallacy that networks are reliable, when, in fact, they are not
▶Thus, a brief revisit to availability themes before continuing
▶The CAP theorem (Brewer 2000) states that any distributed
system can only provide two of the following three guarantees:
1. Consistency: every read always receives either the most
recent write or an error
2. Availability: every request receives a response but without
guarantees that the response contains the most recent data
3. Partition tolerance: a system continues to operate despite
arbitrary partitioning due to network failures

Closed: requests are passed to a service and faults increase
counters. If a counter exceed a threshold, the circuit breaker is
tripped and transitions into the open state.
Open: requests are not passed to a service. Instead, all faults
are passed to clients immediately. If responsiveness is again
gained or ping-echos indicate so, the circuit breaker transitions
into the half-open state.
Half-open: only a limited number of requests are passed to a
service. If a service responds with successes, the circuit
breaker transitions again into the closed state. But if requests
still fail, the circuit breaker transitions back to the open state.

Performance Patterns: Throttling (2/11)
▶In terms of architectural patterns, throttling is usually
implemented with the circuit breaker pattern; thus
1. The closed state is the “normal” state; a circuit breaker is
measuring load against a predefined threshold
2. Throttling occurs in the circuit breaker’s half-open state:
load is limited by graceful degradation
3. The open state is for emergencies: throttling or other things
did not help; the system will return errors to all clients
▶For simplicity, we assume that the threshold is static,
although some algorithms may involve also dynamic thresholds

▶Thus, we may dynamically scale up and down in contexts
such as cloud computing, but there are again trade-offs
• When scaling down, we are reducing the QoS for our clients
