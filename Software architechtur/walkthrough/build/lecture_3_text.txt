A Note on Terminology Again
▶Today and in the subsequent lectures, architectural tactics
refer to high-level design choices, whereas architectural
patterns are smaller and less abstract designs
• This terminology comes from Bass et al. (2021), but, once
again, it is not universally shared – Fairbanks (2020) uses the
term architectural styles in place of architectural tactics
▶One way to think about architectural patterns is to consider
the generalizability of traditional, module-level design patterns
to a level of components and whole systems

Evaluating Integrability (1/6)
▶Bass et al. (2021) suggest evaluating each component Ci in a
system S with two criteria:
1. Size: the number of potential dependencies between the
interfaces of Ci and S;
2. Distance: the difficulty of resolving differences of each of the
dependencies.
▶Of these, size is a rather objective metric because we should
be able to at least approximate a metric such as σ = Pn
i=1 Ci
for a system with n = |S| components
• Yet, a definition for a dependency is needed

===== PAGE 11 =====
Evaluating Integrability (2/6)
▶Latendresse et al. (2022) list the following dependency types:
1. Run-time dependency is needed to run a program
2. Development dependency is needed to develop a program
3. Installed dependency is needed during installation
4. Depth is a dependency’s level in a dependency tree
5. Transitive dependency is a dependency with a depth > 1
6. Context refers to a context in which a dependency is used

Evaluating Integrability (5/6)
▶Thus, we cannot consider only code-level interfaces, which is
also the reason for using the generic connector concept
▶Bass et al. (2021) propose five distance concepts:
1. Syntactic distance: any two connected components via a
connector must agree upon the number and type of data
elements being shared (e.g., versus integers vs. floating points,
XML vs. JSON, machine-readable vs. binary, and so forth)
2. Data semantic distance: any two connected components via
a connector must agree upon the semantics of data elements
shared (e.g., versus seconds vs. nanoseconds, kilometers vs.
miles, Celsius scale vs. Kelvin scale, etc.)
14 / 83

===== PAGE 15 =====
Evaluating Integrability (6/6)
▶The remaining three distance concepts are:
3. Behavioral semantic distance: any two connected
components via a connector must agree upon behavior (e.g.,
versus startup vs. shutdown, a state A vs. a state B, client vs.
server, shared resource used by thread A vs. thread B, etc.)
4. Temporal distance: any two components must agree upon
time (e.g., versus 10 Hz vs. 100 Hz, latency of 100
milliseconds vs. 10 milliseconds, a day vs. a week, etc.)
5. Resource distance: any two components must agree upon
any resources shared (e.g., shared memory, shared CPU time,
shared disk space, shared networking bandwidth, etc.)

Integrability Tactics: Limit (1/13)
▶The most obvious tactic is to avoid a dependency
• The same point applies to a risk-based approach; maybe a
given dependency is just too risky not only in terms
interoperability but also other QAs (e.g., number of other
dependencies it brings, security, vendor lock-in, etc.)?
▶A second tactic is to bundle a dependency into your local tree
instead of managing it via a package manager
• While there are trade-offs, some prefer bundling (Cox 2019)
▶The third tactic involves different ways to encapsulate the
dependency – understood in the present context generally as
an introduction of some connector through which elements
must always communicate

Integrability Tactics: Limit (11/13)
▶Besides potential latency issues, the publish-subscribe
pattern has some other built-in drawbacks, among which are:
1. Usually, a subscriber cannot be sure how long it takes to
receive a published message via a broker sent by a publisher
2. Often, a publisher cannot be sure whether a subscriber has
received a message (unless a broker provides the functionality)
3. Given that messages are typically sent asynchronously,
determinism cannot be usually guaranteed because an
order at which methods are called can vary
4. Testability QA may suffer because testing can be hard
5. Security QA may suffer too because end-to-end encryption
cannot be provided due to publishers and subscribers not
knowing about each other
• However, there is a trade-off because security may improve
due to encapsulation via isolation techniques

Integrability Tactics: Adapt (1/2)
▶A discovery service is a classical architectural pattern; the
examples include the DNS and DHCP protocols
• The idea is similar to the publish-subscribe pattern;
machines discover services without knowledge of each other
▶Tailoring is a tactic that adds or removes capabilities (e.g.,
translations between formats, buffering, computing resource
limits, prepared statements to avoid SQL injections, etc.)
from components without changing them
▶Finally, configuration is similar to tailoring but typically
occurs either at build time (cf. conditional compilation), a
system’s startup (e.g., via configuration files), or runtime
(e.g., via a specific protocol)

Maintenance (3/3)
▶Then, a fundamental choice is about the system’s expected
lifetime; that is, how long will it be maintained?

===== PAGE 50 =====
Modifiability and Changes (2/6)
2. How likely are changes? An estimate is needed, although it
is highly dependent on a system and its environment (cf., e.g.,
“not that likely in the domain of safety-critical systems”)
3. What is likely to change? In theory, everything can change,
but not everything can be changed; hence, decisions are
needed about will be supported and what will not
4. Where are changes likely to be made? At least the
following are possible: source code (e.g., fixing post-release
bugs, etc.), compilation (e.g., conditional compilation and
compiler flags, etc.), configuration (including end-user
configuration of a system), and internal deployment
environment (e.g., a hosting service), among other things
50 / 83

===== PAGE 51 =====
Modifiability and Changes (3/6)
5. Who is likely to make changes? Answers range from
developers to end-users, stakeholders, and third-parties
6. What is the cost to make changes? Note that an answer
covers two dimensions: a cost of making a modification (such
as changing source code) and a cost of designing and
introducing features to make a system more modifiable
• Therefore: even in case improving modifiability during
development-time would incur substantial costs, it may still
decrease the overall costs particularly in case changes are
expected to be frequent

Change Types (1/3)
▶Some frequently encountered change types include the
following four broad categories (Bass et al. 2021):
1. Changes caused by scalability are often encountered; there
will be a lecture on this topic, and, thus, for the present
purposes, it suffices to say that scalability means
“accommodating more of something”
2. Changes caused by not following a principle of location
independence are common too; adhering to the principle
generally means that a location will be easy to modify for a
machine in a distributed system
• In distributed systems many machines and their components
interact but their deployment locations are not often known
prior to a deployment and runtime operations

Change Types (3/3)
4. Changes caused by not having designed a system with

• Thus, one should seek to minimize dependencies to any

▶To recall the basic design principle of object-oriented (OO)
programming: modules should be cohesive and loosely coupled
▶Therefore, whenever a module is not cohesive, meaning that it
deals with some different responsibilities, it should be split
into two or more modules

• But recall the single responsibility principle
▶Although these are common module-level design tactics, they
generalize also to the component-level, as becomes also clear
during the later lectures

Modifiability Tactics: Defer Binding (1/4)
▶The defer binding tactic generally means that one should make
definite choices as late as possible in a development cycle
▶Bass et al. (2021) illustrate this tactic by noting that a
function f (x) that contains an assertion that some b = 1
must hold is less general than a parameterized function f (x, b)
• For instance, f (·) could denote a deployment scenario and x
a given component among a system’s many components
▶With this notation in mind, a sensible recommendation would
be to start developing with something like f (x, b = 1) in mind,
recognizing b = 1 is something that is planned even though it
may change later on during development or deployment

Modifiability Tactics: Defer Binding (2/4)
▶In addition to OO-related ideas, such as using polymorphism,
modifiability can be often increased by using shared
repositories or shared registries
• The sysctl command and the /proc directory in Linux
would be good examples alongside analogous one in Windows
▶That said, the idea of shared registries applies also to
development; both modifiability and reusability can be often
increased by pushing completed artifacts (from requirements
to documentation) in a project to a shared repository and
then pulling artifacts from there in other projects

Modifiability Tactics: Defer Binding (3/4)
▶More generally, modifiability can be increased by designing
modules to allow configuration at runtime via configuration
files, specific services delivering configurations, and flags
• But a software’s runtime has different stages; often, it may
be a good idea to only do start-up time binding, such that a
configuration is only loaded upon a module’s or a
component’s starting time
▶But do not go overboard; something like the Spring
framework for Java is a debatable design in this regard
▶A common alternative is compile-time binding

More General Modifiability Principles
▶Bass et al. (2021) suggest the following general principles:
1. Principle of least surprise: interfaces should behave
consistently and match the caller’s expectations
2. Small interface principle: two interacting interfaces should
exchange as little as possible
3. Don’t repeat yourself principle: avoid offering redundant
ways to achieve the same goal
4. Uniform access principle: do not leak implementation details
to the interface; resources should be available in the same way
regardless how they are implemented
• If there are no consistency requirements, a caller does not
need to know whether a resource is served from a cache or via
a fresh fetch, for instance

Modifiability Patterns: Layering
▶The layering pattern noted already in the first lecture is a
further good choice for increasing modifiability
• When properly designed, changes can be done at lower
layers without changing the upper layers
• Lower levels may also be potentially reused
• The downsides include performance concerns in case calls
traverse many layers; portability concerns regarding the lower
layers; and potentially constraining development in case
developers of upper layers do not (yet) have abstractions and
functionalities implementd at lower levels

Modifiability Patterns: Client-Server
▶The classical client-server pattern is a good choice for
increasing modifiability already because only loose coupling is
present, including with respect to evolution and developing a
given client and a given server implementation
• Also: there is no coupling between clients once deployed
▶The client-server pattern is a sensible choice also for
scalability reasons; as long as the server-side has capacity, the
client-side can continuously scale
▶The drawbacks include dependence on a network and its
protocols as well as potential security concerns, including with
respect to confidentiality, integrity, and availability

Modifiability Patterns: Plugin (1/3)
▶A so-called plugin pattern is also a good choice for
increasing modifiability; the idea simply is that a system’s core
functionality can be modified and increased through plugins
that use a common connector (Bass et al. 2021)
• It is sometimes also known as a micro-kernel pattern
▶As coupling is mostly limited to the given connector, an
obvious benefit is that one team or organization can develop
the core functionality, whereas others (including also
end-users) can develop the plugins
▶Drawbacks include security and privacy concerns; functionality
is increased by executing arbitrary code
