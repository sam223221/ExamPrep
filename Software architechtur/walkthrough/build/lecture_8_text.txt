===== PAGE 16 =====
Safety
▶Safety “is concerned with a system’s ability to avoid straying
into states that cause or lead to damage, injury, or loss of life
to actors in its environment” (Bass et al. 2021)
▶The definition does not say anything about why a system is
entering into a dangerous state; hence, a conventional way to
separate safety from security is that the latter always involves
an actor who intentionally seeks to cause damage or harm
▶Further note that insecurity may cause safety issues, but
the reverse is more difficult to deduce about (or even untrue
according to some people)

Security Tactics: Detect 7→SIEM/SOC (4/4)
▶To address the issues, automation is being increased
▶As discussed by Vakulov (2026), operators of SIEMs/SOCs
have two primary functions: (1) to determine whether events
are malicious, and (2) for true positives, execute appropriate
action (e.g., blocking a host or isolating a machine)
▶Although automation via (1) still carries a lot of false
positives, Vakulov (2026) argues that the workload of humans
can be reduced also via (2) by increasing operational
autonomy (e.g., execution of routine responses), providing
analytical assistance (e.g., event prioritization), and
delegating decisions (e.g., the system provides
recommendations or even executes responses by itself)

Design Principles for Input Validation (1/9)
▶To put output validation aside for a moment, Arce et al.
(2014) recommend to consider the following:
1. Design and use a centralized validation component, but
audit the component carefully and prefer well-known libraries
2. Transform data into a canonical form; be aware of different
encodings (cf. %-encoding in the Web, UTF-8, etc.)
3. Recall that inputs often depend on states
4. Audit all “nearby code” around inputs
5. Prefer strongly typed and memory-safe languages

▶GitHub Actions are a good example about many of the
previous points on input validation; they are development
workflows that can run in response to various events (such as

currently, in 2026, seeking to improve their security

===== PAGE 56 =====
An Example of GitHub Actions (2/6)
▶As described by Ose & Glass (2026), an obvious starting point
for reducing the attack surface is to define who can trigger
workflows and which events are allowed to be triggered
• These questions are about the restrict access tactic
▶Does CoPilot or a LLM from GitHub Apps really need
permissions for triggering all (or even any....) workflows?
▶In terms of security engineering principles, defining these is
about secure defaults, which, in most cases, should follow a
“deny all” principle (“deny everything, then allow particular”)

===== PAGE 57 =====
An Example of GitHub Actions (3/6)
▶As further described by Ose & Glass (2026), the attack
surface of GitHub Actions can also be reduced by increasing
the granularity of permissions and access controls
▶For this to be possible, the trust boundaries must be
explicit; instead of having coarse repository-wide or
organization-wide secrets, these are tied to specific workflows
▶It must be also ensured that secrets are not inherited unless
explicitly allowed (e.g., a secret can be scoped only to a
specific path or a given workflow can has its own secret)
• This connotes with the least privilege principle

===== PAGE 68 =====
Rethinking Agents as Distributed Systems
▶All of the points from the previous three lectures apply
▶For instance, how other agents can detect a crash of an agent
and continue their work without the crashed agent?
▶How about synchronization: if one agent writes to a file
(e.g., fixes a bug) and other agent reads from the file?
▶How about the CAP theorem; when a partitioning occurs,

Input and Privileges (1/8)
▶As already noted, the principle of least privileges means
that a software should only operate with the minimal
privileges it requires for its functioning
• Theoretically, it too can be seen as an attack surface
minimization technique; a restricted process with minimal
privileges can do much less harm than an unrestricted one
▶The principle is particularly important when processing input
from untrusted sources!
▶Note that these principles align with the performance tactic of
bound execution (i.e., limits, caps, thresholds, etc.)

Input and Privileges (2/8)
▶Dropping privileges after a process has started is a common
and old security design technique (Chen et al. 2002)
• It was allegedly first used by Bernstein with qmail in 1997;
thereafter, many Unix/Linux programs and services have been
refactored to drop privileges from the root user
▶However, not all programs and services can entirely drop the
root privilege; therefore, another technique is to separate
privileges by isolating all risky areas to non-root process(es)
• A beautiful example of the technique is Postfix
