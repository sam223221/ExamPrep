Framing Availability (2/6)
▶Availability generally means that a system is available to
serve, use, deliver, or otherwise function whenever needed
• A security perspective to availability adds that the system
should be available for authorized parties only; in addition,
there are again trade-offs between security and reliability
▶From a reliability perspective, availability is related to the
concept of a system failure, which is a deviation of the
system from its specification (Bass et al. 2021)
• Another closely related reliability concept is fault, which can
be defined as the internal or external cause of a given failure

▶However, given the previous definition of a failure as a

Framing Availability (5/6)
▶More specifically, faults can be prevented (cf. good
development practices and tools), removed (cf. fixing bugs),
tolerated (cf. accept these as residual risks), and forecast
• With these broad techniques, a system can achieve
fault-tolerance, which is broadly what resilience is about
▶Relevant questions include (but are not limited to): how to
detect faults and how to prevent these; how frequent faults
may be; what happens in case a fault triggers; how long can a
system be unavailable; how to notify both faults and failures;
and how to recover from these?

Availability Tactics: Detect 7→Sanity Checking (1/2)
▶In general, sanity checking recapitulates what was said the
fourth lecture about assertions; specific operations, states,
inputs, and outputs are verified at runtime for their “sanity”
▶For instance, a machine or a firewall (in reality, a kernel)
receiving a classical (ICMP) echo request packet verifies its
type (8 bits), checksum (16 bits), identifier (16 bits),
sequence number (16 bits), and possibly custom payload
(variable length) too
• If there are mismatches or corruptions, the package is
dropped and possibly logged

Availability Tactics: Detect 7→Sanity Checking (2/2)
▶A further example of sanity checking is cyclic redundancy
check (CRC), which is a classical error-correction technique
used in hardware and networking for detecting alterations
• The term redundancy signifies that a check value is
appended without adding new information
▶If a given check value does not pass a verification check,
corrective action (such as a retry) can be used
▶While being about integrity, CRC checks are not like hashes
• Nor should they be used for security purposes

Availability Tactics: Detect 7→Voting (1/9)
▶Voting (for a lack of a better term) involves comparing
computational outputs from multiple components that should
produce the exact same output, and if they are not producing
the same output, deciding which output to use
▶Examples of voting tactics include:
1. Replication is the simplest technique: components are exact
copies (replicas) of each other, and, hence, if one component
fails, an output from any other component can be used
• In addition to a performance trade-off (due to overhead),
replication cannot protect against design or implementation
errors due to a lack of diversity between the components

Availability Tactics: Detect 7→Voting (2/9)
2. Functional redundancy adds diversification; although
functionally diversified components should still produce the
exact same output for the same input, they may perform also
other function beyond those used for voting
3. Analytical redundancy allows diversity in both inputs and
outputs; for instance, there are multiple ways to compute an
altitude (sensors, radar altimeter, analytical geometry, etc.)
• Sometimes it is a good idea to cast votes among diverse
candidates also in engineering!

Availability Tactics: Detect 7→Exception Detection
▶By exception detection, we go beyond programming languages
▶At least the following can be theoretically distinguished:
1. System exceptions are dependent on hardware and CPUs;
the examples include division by zero and segmentation faults
2. Parameter fences involve placing known patterns (e.g.,
0xDEADBEEF) immediately after a variable-length parameter to
allow runtime detection of overwrites
3. Parameter typing refers to concrete classes implementing
type-length-values specified in an abstract class; it can be used
to enforce type safety even in languages without strong typing
4. Timeout refers to a pattern already elaborated; an exception
(or interrupt, etc.) is raised in case a component detects that
another component has failed to meet its timing requirements

Availability Tactics: Repair 7→Redundancy (1/3)
▶Redundancy, whether through replicated or diversified
components, is the classical repairing tactic
• To recall: if one fails, others can step in
▶However, achieving redundancy even with replicas can be
challenging due to difficulties to ensure data consistency
▶Four consistency types can be enumerated:
1. If high availability is required, eventual consistency can be
guaranteed: whenever there are no updates, all replicated are
merely assumed to eventually converge to the same data state

Availability Tactics: Repair 7→Redundancy (2/3)
2. Quorum consistency resembles the earlier take on majority
voting: an update succeeds only in case a quorum (such a
majority or 1/3:th of components) acknowledges the update
• Something akin the Observer pattern could be used
• Although polling is often discouraged – and often for a
good reason, it too might be suitable in some contexts
3. Synchronized consistency allows simultaneous writes at
multiple data components but requires conflict resolution to
handle concurrent updates
4. Read-after-write consistency means that all subsequent

Availability Tactics: Repair 7→Retry (2/3)
▶The previous simple retry example could be improved with
exponential back-off, meaning that the sleeping delay
between calls would increase as failures increase
▶Another improvement might be a circuit breaker pattern:
instead of returning FAULT after max retries is reached, the
routine would wait longer and then try the loop again,
perhaps only returning FAULT after a few such iterations
• To avoid wasting resources, the rationale is to fail fast
▶Sometimes it also a good idea to add timeouts to all
availability-specific calls (memory, networking, etc.)
▶Caching is a good idea too; it is covered in the next lecture

Availability Tactics: Repair 7→Degrade and Reconfigure
▶The graceful degradation tactic seeks to maintain the
availability of most critical functions while degrading,
suspending, or outright dropping less critical functions
• For instance, in the face of a denial of service attack against
a public transportation system, a payment system might be
maintained at the expense of maps and related auxiliary things
▶However, it is possible to also use the graceful degradation
without ties to critical functions; e.g., serve the majority by
denying requests for the minority; only serve some particular
geographic locations; compute work in progress first without
accepting new requests; etc.

Availability Tactics: Repair 7→Reconfigure
▶The reconfiguration tactic is about recovering from failures

• For instance, a server under denial of service might reload a
configuration that drops HTTP but still allows SSH
▶A good example would be a reconfiguration that slows things
down once some critical threshold is reached
• For instance, CPUs throttle them in case of overheating

Availability Tactics: Prevent 7→Isolate (1/4)
▶Isolation is a general tactic for both security and availability
▶Recall the notion of blast radius from the fourth lecture
• If machines and their components, or some of them, are
isolated from each other, a blast radius may not cover the
entire system; important machines or components may still
work even in case most are down
• A concept of sharding is often used in this context
▶The blast radius concept correlates with a more general
concept of cascading failures; a single failure somewhere in a
large system may – in the worst case scenario – cascade into
the whole system, possibly taking it down

Availability Tactics: Prevent 7→Isolate (3/4)
▶When designing isolation, Adkins et al. (2020) recommend
functional separation of different failure domains
• The goal is to minimize a probability that a failure does not
affect all components of a system at once
▶They further emphasize data isolation; each failure domain
needs its own data to be functionally independent
• The goal is to minimize the propagation of corrupted or
otherwise bad data from one failure domain to another
• Note that “data” should be understood broadly here; it is a
good idea to write also known good configurations to disk

with fault-tolerance functionality is a good idea
▶For instance, a poorly designed module needing a network
might throw an exception (or worse) in case a given network
happens to be down
• By slightly increasing competence of the module, it could be
redesigned to put itself into an idle state and wait for the
network to come back
▶But recall also the earlier point that sometimes it is better to
also crash and burn loudly; hence, there are again trade-offs

Availability Patterns: FIFO and LIFO Queueing
▶Sometimes, a LIFO (“last in, first out”) queuing is better
than FIFO (“first in, first out”) queuing
• A FIFO processes requests as they arrive, whereas LIFO
prioritizes more recent requests
• For instance, using a FIFO can be useful to avoid scenarios
where a large amount of clients are about to retry their
requests; the retries themselves eat up resources even more

Availability Patterns: Dependencies
▶Adkins et al. (2020) recommend consider dependencies when
designing for availability (here, recall the points from the third
lecture about different dependency concepts)
▶For instance, redundancy could be engineered via isolating
critical but low-dependency components; these are those
particular components that have only minimal dependencies
to failure domains of a system
▶In addition, they give two practical advices:
1. “Use data cached on local storage to avoid depending on a
remote database”(cf. versus Supabase, anyone...?)
2. With some components, deploy first with “older code and
configs to avoid recent bugs in newer versions.”

Availability Patterns: Taints and Quarantines
▶A further pattern, which is related also to security, is about
tainting: whenever a corruption is detected, the corrupted
data should not be used and no actions should be derived
from the corrupted data
▶For these reasons too, the data isolation is important
▶Yet, it is a good idea to quarantine the corrupted data
• This is what for instance file systems and hard drives do

===== PAGE 72 =====
Availability Patterns: Saga (1/3)
▶The saga pattern divides transaction processing into local
transactions: each local transaction completes its action
atomically within a single component (or service)
• It is similar to the batch-sequential pattern from the third
lecture but with an addition of error correction
▶Each local transaction updates its own database and notifies
the subsequent local transaction about its completion
▶The pattern is good for maintaining consistency of data, but
complexity increases because if a local transaction fails,
corrective transactions must be propagated

The Maintainability and Cost Trade-Offs
“Today, systems often degrade without clear signals. Traces look fine,
dashboards stay green, and yet performance quietly erodes. This shift is
not just about scale. It reflects how modern systems accumulate
complexity long before they actually need it.”

the operational cost of running the system today.”
