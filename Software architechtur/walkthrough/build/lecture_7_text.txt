Kubernetes: Scaling (1/7)
▶There are two analytical ways to frame and understand scaling
of Kubernetes clusters:
1. Horizontal scaling means that new machines or new pods
(replicas) are added to meet increased load
2. Vertical scaling means that more resources (memory, CPUs,
etc.) are added to existing machines to meet increased load
▶It is difficult to interpret what these might mean in terms of
the Amdahl’s law, but, nevertheless, diminishing returns seem
a reasonable assumption at some point (cf. also costs)
▶By default, Kubernetes (2024a) comes with an autoscaler that
also scales down horizontal or vertical upward scaling once
load has decreased

Kubernetes: Scaling (3/7)
▶Most of the autoscalers use different metrics (such as CPU
usage) for determining appropriate upward scaling
▶They also provide minimums (e.g., 10 machines with 100

• It is a good idea to define these to avoid cost surprises
▶Given that the autoscalers are dynamic algorithms, Google

times because both add overhead
▶While logging is necessary (and also for monitoring a cluster),
also these add to costs when done excessively in huge clusters

Load Balancing Revisited: DNS (1/6)
▶The previous examples about scaling Kubernetes clusters are
essentially about dynamic algorithms that are common also
in more general-purpose load balancing
• The throttling and scheduling algorithms from the previous
lecture are also good examples of dynamic algorithms
▶In contrast, static algorithms distribute load across systems
without taking into account their current states or health
▶The Domain Name System (DNS) is a classical example still
partially but fundamentally relying on static algorithms; many
(but not all!) Internet servers distribute via name servers
addresses to clients in rotation using round-robin scheduling

Load Balancing Revisited: DNS (3/6)
▶While simple, DNS-based round-robin scheduling provides
scalability because clients are routed to different servers
• Although such round-robin scheduling continues to route
requests also to dead or otherwise non-functioning servers
(because there are no heartbeats or other monitoring), it does
still provide some coarse fail-over for clients because they will
pick a subsequent address (server) after a timeout
• In contrast, the active-active and active-passive (“standby”)
load-balancing scenarios from the fifth lecture require
heartbeats, polling, or other monitoring
▶Some DNS functionality, such as for email, use more complex
but still static (‘or “semi-static”) algorithms

▶OpenStack (2023) can also be used for simple load-balancing
at the topmost, seventh (application) layer in the OSI model
▶There are three policy actions for HTTP:
1. REJECT: request is denied
2. REDIRECT TO URL: request is redirected to a given URL
3. REDIRECT TO POOL: request is redirected to a given pool
▶Above, the term “pool” is a shorthand for one or more
back-end servers, including web servers in particular
▶Among the rule types are HEADER (matches fields in HTTP
request headers), PATH (matches HTTP request URIs), and
FILE TYPE (matches specific file types, such as .png)

▶Analogously to the previous lecture on performance, at a
high-level of abstraction, we can think about “time
scalability” (or load scalability) and “space scalability”
▶However, these are related: with big data, the sizes of objects
are large to transmit via a network (increases a), difficult to
compute due to memory constraints (increases b), and slow to
read and write (increases c)
▶With this terminology, we can “regard a system as having
space-time scalability if it continues to function gracefully as
the number of objects it encompasses increases by orders of
magnitude” (Bondi 2000, p. 3; boldfacing added)

Performance and Scalability Diagnosis (1/4)
▶Bodi (2000) also gives some practical diagnosis guidelines:
1. Identify and diagnose all cases in which components—or even
a whole system—could enter into some absorbing state (such
as a deadlock) or a wholesale failure
2. Identify a system’s parts that can run in parallel
3. Identify self-expanding performance bottlenecks
(cf. cascading failures but from a performance viewpoint)
4. Identify and eliminate unproductive execution cycles
5. Seek to minimize temporary execution cycles
6. Identify whether scalability problems are due to scheduling
