===== PAGE 10 =====
A Note on the Course’s Progression
▶Traditionally and internationally, there have been two primary
pedagogical ways to teach a course on software architecture:
the “traditional one” (from designs to implementations and
operations) and the “reverse one” (from operations and
implementations through reverse engineering to designs)
▶However, for this course, I follow a mixed approach, starting
from “supposedly somewhat familiar” topics and slowly
progressing toward “supposedly rather unfamiliar” topics,
incrementally including also rarely covered yet I believe
interesting topics from a software engineering perspective
• That said, do not rely on this topical overall schedule for
strategic grading – as said, it will be a non-linear mixture of
topics, all of which you should master for a good grade

===== PAGE 21 =====
An Example Definition
▶Clements et al. (2010, quoted from Fairbanks 2010, p. 16)
provides the following definition: a “software architecture of a
computing system is the set of structures needed to reason
about the system, which comprise software elements,
relations among them, and properties of both”
▶Although terminology varies slightly from an author to
another and from a standard to another, the definition
contains the basic ingredients with which the lectures operate:
systems, structures, elements, relations, and properties
• A little more precise definitions are provided later (on p. 39)

===== PAGE 22 =====
Six Truisms (adopted from Bass et al. 2021)
1. A software architecture is an abstraction
2. A software architecture is about engineering and not art
3. A software architecture is software design, but not all software
designs are about software architectures
4. Every software system has an architecture
5. Not all software architectures are good
6. There are often (but not necessarily always) both
requirements and constraint for software architectures, which
further usually require some domain expertise

===== PAGE 29 =====
Software Architecture and Detailed Design
“Defining architecture as the set of structures needed to reason
about a system is useful in that it focuses attention on the purpose
of the architecture (reasoning), but problematic in that it blurs the
line between architecture and detailed design. It is simpler and
more clear to say that architecture is the macroscopic parts of the
design, such as modules and how they are connected, leaving
detailed design to cover everything else.” (Fairbanks 2010, p. 17)

===== PAGE 31 =====
Quality Attributes for Software Architectures (2/3)
▶Thus, when designing a software architecture (but not the
whole implementation), one seeks to fulfill quality attributes
• Although these are usually dependent on non-functional
requirements, constraints, and domain knowledge,
functionality itself (i.e., functional requirements) matters less
• For instance, a customer might have a non-functional
requirement that availability is of utmost importance, and,
hence, a software architect’s task would then be to guarantee
availability via a suitable architecture under given constraints
(such as hardware, financial resources, potential data
localization requirements for on-premise deployments, etc.)
31 / 84


===== PAGE 32 =====
Quality Attributes for Software Architectures (3/3)
▶Like many (but not all) conventional requirements, also
quality attributes should be prioritized; hence, in the previous
example availability should be the top-ranked attribute (due
to “utmost importance”) even though we would still want,
say, maintainability with a some priority as a quality attribute
▶Therefore, an evaluation of an architecture is best done by
considering the prioritized quality attributes
▶A good architecture allows incremental implementation to
avoid a need to integrate everything at once
▶However, an architecture — whether good or bad – also
always constraints the implementation

Requirements and Software Architecture (4/4)
▶When considering the typical “mountain range” twin peaks
model, the left-hand side usually contains many non-functional
requirements that should be transferred to quality attributes
▶Regarding the right-hand side, however, it is usually
recommended to enhance architectural reuse (including via
tactics and patterns) by minimizing design alternatives and
their vocabularies (two or three might be more than enough)
▶In addition, keeping it simple is always a good idea,
including with respect to documentation, communication with
stakeholders, confidence, interoperability, and shorter selection
time between the design alternatives sketched

Terminology (1/4)
▶Although there exists no universally accepted terminology for
software architectures, some key concepts are still shared
▶For instance, Bass et al. (2021) separate (a1) components,
(a2) connectors, (a3) modules, and (a4) allocation structures,
whereas Lano & Tehrani (2023) separate (b1) components,
(b2) connectors, and (b3) configurations
• In what follows, the terminology adopted is based on a
mixture of their concepts with some clarifying framings

===== PAGE 39 =====
Terminology (2/4)
▶In this course, the following ten concepts are used:
1. Module is a large unit of code; Java’s packages are the closest
reference point in terms of programming languages
2. Interface connects two or more modules
3. Component is a large collection of source code modules and
other elements in a software stack from servers to databases
4. Process is a Unix-like operating system (OS) process
5. Machine is a physical or virtual machine running an OS
6. System contains one or more machines with components
7. Deployment is a system that is running in production
8. Environment is a surrounding for a deployment
9. Element is either a module, a component, a process, a
machine, a system, or a deployment
10. Connector connects elements via established channels

▶Do not expect a 1:1 correspondence between components and
modules; many components may be running in parallel even

• However, in this course we are interested more in large
distributed systems whose deployments may operate with
even millions of machines
▶The term connector is a catch-all capturing everything
(including interfaces) with which elements talk to each other

Structures
▶Bass et al. (2021) use the term structure as a catch-all for
connecting elements via relations (i.e., connectors)
▶Even though we do not operate at the level of hardware,
the previous two examples demonstrate that even the central
processing unit (CPU) level can be conceptualized as a
structure via the same terminology and visual notation
▶Most of the time we will operate at the component-level
because it gives the most flexibility to describe systems
• The module-level is reserved for the rare cases when we
operate at the level of code and programming languages

===== PAGE 45 =====
Component Structures
▶With component structures, we are interested in fundamental
questions such as: what are the major executing components
and how do they interact; which parts of a system can be
replicated; how does data process through a system; and are
there opportunities for parallel or concurrent execution?
▶Thus, component structures are key units of an architecture
▶They should be cohesive, use encapsulation, allow
reusability, substitution, and independent deployment,
and they should generally be composable into larger systems
• Note that the two previous low-level CPU examples already
used these concepts, including encapsulation

The Substitution Principle (1/6)
▶“Component B can be substituted for component A if:
1. A’s provided interfaces are preserved by B (B has the same or
more provided interfaces than A)
2. The signature and expected behaviour of all operations of all
provided interfaces of A are preserved by B
3. B’s required interfaces are the same as or fewer than A’s – B
cannot require more functionalities than A did, because the
existing system context (of A) may not provide these.”
(a direct quotation from Lano & Tehrani 2023, p. 26)

The Substitution Principle (5/6)
▶Liskov’s classical substitution principle generally means that if
S is a sub-type of type T, then objects of type T can be
replaced with objects of sub-type S without alterations
• See Liskov & Wing (1994) for a classical take
▶Lano & Tehrani (2023) extend the Liskov’s substitution
principle to components and interfaces; in essence, any given
implementation of any given interface should be entirely
consistent with the interface’s specification

Cohesion and Non-Cohesion (2/2)
▶The previous example is not only about cohesion (or a lack
thereof) but also about the substitution principle because the
provided marketing module requires an interface to Google
▶Therefore, Bass et al. (2021) further emphasize that a
software architecture should never depend on a particular
vendor and its products, tools, and their versions
• If a dependence is a must to a particular vendor (such as
Google), an architecture should be designed so that changing
to different versions of the vendor’s tools and their versions
should be straightforward and inexpensive
• Thus, an architecture should generally avoid vendor lock-in

===== PAGE 63 =====
The Segregation Principle (1/2)
▶The segregation principle generally means that components
(and their modules) should not depend on components
and connectors (and interfaces) they do not actually use
• Unnecessary connectors (i.e., dependencies) signal wasted
development effort and wasted resources when a system is
deployed (in addition to other, such as legal, concerns)
▶For in-house architectures, autonomous components are often
preferable albeit with a trade-off that they may incur bloat as
many components may implement the same functionality
▶For suppliers of components, a fine-graded segregation of
connectors (interfaces) is preferable over a single connector
(interface) that would be forced upon all users

The Single Responsibility and Open-Closed Principles
▶Lano & Tehrani (2023) outline two additional general-purpose
principles for component structures:
1. The single responsibility principle: each component should
only have one reason to change, as ensured by a single
responsibility carried out by each component
2. The open-closed principle: whenever possible, components
should be changed only by adding new functionality and not by
modifying existing functionality
▶When contemplating these principles, the running example of
reserving flights and hotel rooms is arguably ill-designed to
begin with; it should likely be the flight and hotel room

Module Structures (1/5)
▶Module structures partition a system into modules, that is,
implementation units, including at a level of source code
• Hence, modules may range from layers and packages all the
way to classes, files, functions, and variables
▶In terms of interfaces between modules, one can think in

“is a part of” relations) and in terms of “has a”

▶Class diagrams – which, however, are not covered in this
course – are a classical example of providing views to concrete
implementations of modules

Module Structures (5/5)
▶The previous na¨ıve ensemble learning architecture conveys the
wisdom from Bass et al. (2021) that components (and their
modules) that produce or store data should be separate from
components (and their modules) that consume data
• This separation improves modifiability because changes can
be done at either side; for instance, it is much easier to add a
new ML algorithm to the ML component because no changes
are required at the database component
• If both components still require changes (e.g., new data),
the changes can be done incrementally via a staged upgrade
